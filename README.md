### *Authentication System*
Система аутентификации и разграничения прав доступа (RBAC), построенная «с нуля» без 
использования встроенных механизмов Django (sessions, admin, groups).
Система позволяет гибко настраивать доступ к системным и бизнес-объектам через базу данных, 
разделяя права на «свои» (владение) и «все» (глобальный доступ).

### *Используемые технологии*
1. Django + DRF - базовый фреймворк (используется как каркас)
2. Poetry - менеджер для управления зависимостями и окружением
3. PostgreSQL - реляционная СУБД, используемая для хранения кастомной ролевой модели
4. jwt (PyJWT) - библиотека, использующаяся для создания и декодирования токенов
5. bcrypt - библиотека для хеширования паролей

### *Схема базы данных*
#### User
Кастомная модель пользователя в системе. Каждому пользователю может быть назначено несколько
ролей. Операция удаления пользователя не удаляет его из базы, а помечает неактивным (флаг
is_active = False).
В целях безопасности предусмотрено наличие системного пользователя, который имеет флаг
is_system = True. Он имеет права на все объекты, эти права зашиты в коде, что предотвращает 
блокировку системы даже при очистке БД. Его нельзя удалить или изменить через API.
- `id` - первичный ключ
- `email` - адрес электронной почты (уникальный)
- `password` - пароль
- `full_name` - полное имя
- `is_active` - признак активного пользователя
- `is_system` - признак системного пользователя
- `created_at` - дата создания
- `roles` - роли в проекте (связь many-to-many)

#### Role
Описывает возможные пользовательские роли в проекте. Пользователь, не выполнивший вход в 
систему, считается гостем и обладает всеми ролями, у которых стоит флаг is_guest (зашито в
коде). Все роли с флагом is_default_role автоматически привязываются к новому пользователю 
в процессе создания учетной записи.
- `id` - первичный ключ
- `name` - наименование
- `slug` - слаг
- `is_guest` - признак гостевой роли
- `is_default_role` - признак роли, добавляемой по умолчанию после регистрации

#### BusinessElement
Описывает возможные виды ресурсов в проекте, к которым будет осуществляться доступ. Это могут 
быть бизнес-объекты и системные таблицы (пользователи, роли, ресурсы и таблица настройки 
прав).
- `id` - первичный ключ
- `name` - наименование
- `slug` - слаг
- `is_system` - признак системного ресурса

#### AccessRule
Описывает матрицу прав. Права даются на просмотр (GET), создание (POST), обновление (PATCH)
и удаление (DELETE) конкретных видов ресурсов для конкретной роли. Делятся на две категории:
права на все объекты (с префиксом _all) и права на собственные объекты, то есть созданные 
текущим пользователем. Право на создание логично имеется в единственном экземпляре.
Доступ к бизнес-объектам осуществляется согласно данной таблице с учетом ID создателя объекта,
прописанного в поле owner_id объекта. Доступ к системным ресурсам учитывает только права на 
все записи в таблице (или create). Гостевые роли также учитывают только общие правила (create
либо права с префиксом _all). Системный пользователь имеет все права по умолчанию в обход 
данной таблицы.
- `id` - первичный ключ
- `role` - ID роли
- `element` - ID ресурса
- `read_permission` - право на просмотр собственных объектов
- `read_all_permission` - право на просмотр всех объектов
- `create_permission` - право на создание объектов
- `update_permission` - право на обновление собственных объектов
- `update_all_permission` - право на обновление всех объектов
- `delete_permission` - право на удаление собственных объектов
- `delete_all_permission` - право на удаление всех объектов

#### BlacklistedToken
Черный список - механизм аннулирования токенов при выходе пользователя из системы.
Создан в целях безопасности, чтобы гарантировать закрытие сессии с пользователем по
токену (пользователь не будет авторизован).
Предполагается, что для оптимизации хранилища предусмотрена очистка таблицы 
BlacklistedToken по истечении срока жизни токена (возможно, с помощью фоновой задачи).
- `id` - первичный ключ
- `token` - jwt токен
- `expires_at` - срок истечения токена
- `blacklisted_at` - дата помещения токена в черный список

### *Базовая конфигурация системы*
Для первоначальной настройки системы прав и в целях демонстрации работающей системы 
реализована команда setup_admin, которая наполняет базу тестовыми данными, а также создает 
суперпользователя. После выполнения этой команды таблицы будут содержать следующие данные:

#### Role:
1. Администратор 
2. Менеджер
3. Пользователь (роль по умолчанию)
4. Гость (гостевая роль)

#### BusinessElement
Системные ресурсы:
1. Пользователи
2. Роли
3. Правила доступа
4. Бизнес-элементы
Бизнес-объекты:
1. Товары
2. Магазины
3. Заказы

#### AccessRule
1. Администратор имеет все права доступа к системным ресурсам, кроме создания пользователей.
2. Менеджер имеет права на просмотр всех пользователей и всех заказов, а также права 
создания, редактирования и удаления всех товаров и всех магазинов.
3. Пользователь может создавать новые заказы, просматривать, обновлять и удалять свои заказы.
Также у него есть права на просмотр всех товаров и магазинов.
4. Гость может смотреть все товары и магазины.

Дальнейшая конфигурация системы осуществляется системным пользователем, а затем - 
пользователями с ролями "Администратор" и "Менеджер". 
Создание пользователей происходит только в процессе регистрации.

### *API endpoints (префикс: /api/)*
### `POST` `/auth/register/`
Регистрация пользователя. Назначение ему прав по умолчанию.
Необходимо указать адрес почты, имя и пароль.

### `POST` `/auth/login/`
Вход в систему (аутентификация). Возвращает access-токен типа Bearer, который нужно будет 
передавать каждый раз в заголовке запроса к защищенному ресурсу.

### `POST` `/auth/logout/`
Выход из системы. Токен заносится в черный список.

### `GET/PATCH/DELETE` `/auth/me/`
Управление личным профилем (просмотр, обновление, удаление). Адрес почты и пароль поменять 
нельзя. Удаление происходит мягко, запись в БД остается с флагом is_active = False.
Недоступно для системного пользователя.

### `GET` `/auth/users/`
Просмотр списка пользователей.

### `GET/PATCH/DELETE` `/auth/users/<pk>/`
Просмотр, обновление и мягкое удаление пользователя с ID=pk в случае если пользователь - не
системный.

### `GET/POST` `/manage/rules/`
Просмотр списка ролей, создание новой роли.

### `GET/PATCH/DELETE` `/manage/rules/<pk>/`
Просмотр, обновление и удаление роли с ID=pk.

### `GET/POST` `/manage/elements/`
Просмотр списка элементов, к которым осуществляется доступ, создание нового элемента.

### `GET/PATCH/DELETE` `/manage/elements/<pk>/`
Просмотр, обновление и удаление элемента с ID=pk.

### `GET/POST` `/manage/access_rules/`
Просмотр списка правил, создание нового правила для определенной роли и элемента.

### `GET/PATCH/DELETE` `/manage/access_rules/<pk>/`
Просмотр, обновление и удаление правила с ID=pk.

### `GET/POST` `/mock/orders/`
Эмуляция эндпоинта для просмотра списка доступных заказов и создания нового заказа.
В случае установленного права read_all_permission выдает список всех заказов, иначе 
если установлено только право read_permission, фильтрует заказы по owner_id и 
выдает список только своих заказов.

### `GET/PATCH/DELETE` `/mock/orders/<pk>/`
Mock для просмотра, обновления и удаления заказа с ID=pk. Доступно, если установлено 
соответствующее право с префиксом _all или установлено право на свои заказы и это
свой заказ.

### `GET/POST` `/mock/products/`
Эмуляция эндпоинта для просмотра списка доступных товаров и создания нового товара.
Алгоритм выдачи товаров аналогичен эндпоинтам заказов, с учетом системы прав.

### `GET/PATCH/DELETE` `/mock/products/<pk>/`
Mock для просмотра, обновления и удаления товара с ID=pk. Работает аналогично эндпоинтам
заказов.

### *Особенности реализации*
1. Логика хеширования и проверки пароля вынесена в сервисный слой - файл crypt.py. В текущей
реализации используется bcrypt, но она может быть заменена впоследствии любым другим
инструментом.
2. Аналогично логика аутентификации реализована в отдельном сервисе в файле auth.py. 
Инверсия зависимостей позволяет сменить механизм JWT на любой другой (например, OAuth2 или
Session-based) без изменения логики представлений и authenticators.
AuthService - интерфейс, описывающий общие методы работы с пользовательскими сессиями
и учетными данными. Его реализация доступна через объект auth_service.
JWTAuthService - текущая реализация аутентификации на основе JWT токенов. Сейчас 
используются долгоживущие access-токены, но в будущем планируется использование пары
токенов access/refresh.
3. Для тестирования доступа к бизнес-объектам применяются Mock-View, которые по обращениям
к ним выдают список потенциальных объектов.
4. Проверка доступности ресурса выполняется в кастомном permission RBACPermission в два этапа:
- глобальный (has_permission) - проверяет наличие прав на тип ресурса в целом (эндпоинт);
- объектный (has_object_permission) - срабатывает при обращении к конкретной записи.
Если доступа к ресурсу нет и пользователь не определен в системе, выдается ошибка 401. 
Если пользователь определен, но запрашиваемый ресурс ему не доступен, — выдается 
ошибка 403 Forbidden.

### *Запуск и настройка*
1. Настроить переменные окружения в файле .env для доступа к базе данных PostgreSQL:
- *POSTGRES_DB* - имя базы данных Postgres
- *POSTGRES_USER* - имя пользователя Postgres
- *POSTGRES_PASSWORD* - пароль пользователя Postgres
- *POSTGRES_HOST* - имя хоста/адрес сервера
- *POSTGRES_PORT* - порт

2. Выполнить миграции базы данных с помощью команды:
```bash
poetry run python manage.py migrate
```
3. Выполнить инициализацию системы (создание ролей, прав и Root-админа) с помощью 
команды настройки:
```bash
poetry run python manage.py setup_admin
```
4. Запустить сервер:
```bash
poetry run python manage.py runserver
```
